# Product

## Register

product

## Users

Chorus is for people who use AI as a daily working surface: builders, writers, researchers, founders, and technical users who want a private local chat history and direct control over provider keys. The iOS version is for the same user away from the Mac, usually in short sessions where they need one reliable model, quick recall, and a quiet interface that does not fight the keyboard.

## Product Purpose

Chorus lets a user chat with LLMs using their own API keys while keeping conversations local. The desktop app compares multiple models and exposes desktop tools; the first iOS product should preserve the Chorus look, message format, settings vocabulary, and local history while focusing on one OpenRouter-backed chat flow.

Success for the iOS MVP means a user can add an OpenRouter key, choose or keep a default model, start chats, stream responses, revisit local history, edit the system prompt and key settings, and feel like they are using Chorus rather than a generic mobile chat clone.

## Brand Personality

Quiet, capable, slightly nerdy. The interface should feel like a compact professional tool: warm but not cute, technical but not harsh, familiar but not disposable.

## Anti-references

Avoid generic consumer chatbot gloss: oversized bubbles, loud gradients, mascot-like onboarding, full-screen marketing hero language, and decorative animation. Avoid porting desktop-only affordances directly to iPhone: hover-only controls, global shortcut copy, resizable split panes, tray/window terminology, shell/MCP controls, and broad file-system permissions in the first release.

## Design Principles

1. Preserve the Chorus atmosphere: small type, restrained warm neutrals, light borders, compact controls, and readable markdown.
2. Design for one-handed mobile chat first: history, messages, composer, model/settings access, and keyboard behavior must feel native on iPhone.
3. Keep trust visible: local history, bring-your-own-key, clear error states, and secure key storage are core product features.
4. Cut desktop power gracefully: omit ambient shortcuts, panels, MCP/tools, screen capture, and multi-model compare until the mobile core is stable.
5. Prefer ordinary product affordances: standard tabs, sheets, lists, buttons, toggles, and forms over custom decorative interaction.

## Accessibility & Inclusion

Target WCAG AA contrast for text and controls. Preserve system light/dark behavior, support reduced motion, maintain touch targets appropriate for iPhone, and avoid relying on hover, keyboard shortcuts, or color alone for core actions.
